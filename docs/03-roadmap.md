# 03 功能路线图

[← 返回文档中心](./README.md)

> 阶段 0～8 的功能计划。**这一版按"护城河优先"重排过**：差异化（crypto / eval / 安全深度）不再堆到最后，而是靠一个早期的垂直切片提前落地。

---

## 一条压倒性的原则

**差异化要早落地，不能全堆最后。**

上一版路线把 crypto、eval、深度安全全排到阶段 7-8。对一个求职作品，这是最经典的翻车形态：通用的 80% 吃光时间，让你"不是又一个 Claude Code 克隆"的那 20% 永远没轮到。所以这一版的锚点是下面这个里程碑——

### 🎯 里程碑 A（目标：约第 6 周）——一个能演示的"诚实垂直切片"

> 手上永远要有一个**完整、诚实、能讲**的端到端故事，而不是九个半成品。一旦求职窗口提前，交得出去。

达成标准（缺一不可）：

- ✅ agent 内核能跑多步任务（已完成）
- ✅ tracing 挂进真链路，一次请求看得到完整 trace
- ✅ 一个**最小 eval 集**（5-10 条），能跑出真实通过率
- ✅ **一个只读 crypto 工具**（如查链上余额 / gas），带确认与审计
- ✅ README 骨架 + 一段 demo GIF

达成后，剩余阶段都是在这个"能演示的核"上加厚，而不是从零攒。

## 进度总览（重排后）

| 阶段 | 主题 | 状态 | 诚实体量 |
|------|------|------|----------|
| **0** | Agent 内核 baseline | ✅ 完成 | — |
| **1** | 可观测 / Tracing | 🔄 进行中 | 1 周 |
| **2** | 终端体验 + 循环控制 + **生产级健壮性** | ⬜ | 2 周 |
| **3** | **最小 Eval harness（回归基线）** ⭐ | ⬜ | 1 周 |
| **4** | 工具补齐 + **差异化纵切**（crypto/MCP）⭐ | ⬜ | 2 周 → 里程碑 A |
| **5** | 智能上下文 + 会话持久化 | ⬜ | 2 周 |
| **6** | **Crypto 场景安全 + 真沙箱** ⭐ | ⬜ | 2 周 |
| **7** | Loop 工程 | ⬜ | 2 周 |
| **8** | 开源门面（+ 可选多 provider） | ⬜ | 增量维护 |

⭐ = 差异化（护城河）。注意它们分散在 3/4/6，而不是全挤在最后。

**三条开发铁律**：

1. **差异化早落地**：每个护城河能力都要有一条能早点 demo 的纵切，别攒到最后。
2. **核心自研、通用用开源**，且能讲清选型。
3. **每阶段结束**：中文 commit + 能复述"做了什么 / why / 选型" + **跑一遍 eval 看有没有掉分**。

## 阶段 0：Agent 内核 baseline 【✅ 完成】

| 模块 | 文件 | 能力 |
|------|------|------|
| API 客户端 | `client.py` | DeepSeek OpenAI 兼容调用 |
| 入口 + 循环 | `main.py` | REPL + `run_turn` tool loop |
| 工具 | `tools.py` | 5 工具 + schema + handler |
| 上下文 | `context.py` | 滑窗裁剪，保留 system |
| 安全 | `security.py` | 确认、黑名单、审计（**baseline，见下方诚实说明**） |
| 提示词 | `prompts.py` | system prompt |

### 面试话术（诚实版）

> 我从零实现了 harness 的心脏：外层 REPL + 内层 tool loop，assistant/tool 消息严格配对、支持并行 tool_calls、错误作为 observation 回传让 agent 自修正。安全目前是 **baseline**——确认 + 路径沙盒 + 审计 + 一个玩具级命令黑名单；真正的"最小权限 / 注入隔离 / 沙箱"是我阶段 6 要做深的。

> **为什么不吹"纵深 5 层"**：`security.py` 现在就是几条子串黑名单 + y/n 确认。面试官一句"给我看"就穿帮。诚实说"这层是真的、那层还薄"，对资深候选人反而是更强的信号——参见 [04-learning-plan](./04-learning-plan.md) 的"自评校准"与"失败模式"两节。

## 阶段 1：可观测 / Tracing 【🔄 当前】

### 目标

一次用户请求 → 完整执行链路可见（每步 LLM / tool、耗时、输入输出）。

### 为什么优先

后面挂 eval、对外演示、调试 agent 行为，全依赖"能看清发生了什么"。**这是阶段 3 eval 的地基。**

### 关键改动

- `tracing.py` 已实现 → **挂进 `main.py` 真链路**：每次 `chat_with_tools` 一个 llm span、每个 tool call 一个 tool span，轮次结束 `finish_trace` + `print_trace`。
- 先跑通 tool span，再加 LLM span + token 统计（`response.usage`）。Langfuse / JSON 导出往后放。

### 验收

- [ ] 跑"读文件 + 修改"多步任务，终端看到完整 trace
- [ ] 每个 span 有耗时、截断后的 input/output、成功/失败状态

### 自研 vs 开源

| 自研 | 开源 |
|------|------|
| 记什么、何时插桩、Trace/Span 结构 | Langfuse 存储、UI、检索（类比 OTEL SDK vs 后端） |

## 阶段 2：终端体验 + 循环控制 + 生产级健壮性

### 目标

用起来像成熟工具，而不是"能跑的脚本"；**并把你十几年 DevOps 存量翻译成 agent 语境的差异化**。

### 功能清单

| 功能 | 说明 |
|------|------|
| Streaming | `stream=True` 逐 token 打印；正确重组分片到达的 tool_calls（按 index 累积拼 JSON——最大的坑） |
| `attempt_completion` | 模型显式 signal 任务完成，harness 决定信不信 |
| 斜杠命令 | `/q` 退出、`/n` 新会话、`/h` 帮助 |
| 循环控制升级 | 修 max_turns 边界（达上限别 append `content=None`）、重复调用检测、达上限时告知用户+给模型收尾机会 |
| **生产级健壮性** ⭐ | **API 重试/超时/限流退避、畸形 tool_call 参数兜底、`input()` 处理 Ctrl-C/Ctrl-D、单次调用失败不杀会话** |

### 为什么"生产级健壮性"是你的差异化

纯 AI 背景的候选人最缺"把 agent 跑到生产、扛住网络抖动"的直觉，而这正是你的本行。一句"我把生产级容错带进了 agent 循环"，别人抄不来。

### 面试话术

> Streaming 不只是体验，是架构问题——用户更早看到输出、tool 执行时能给状态反馈；而流式 + 工具调用交织（tool_call 分片重组）是 harness 的经典难题。容错这块我是按后端服务的标准做的：重试要考虑 tool loop 中间失败时的消息一致性。

## 阶段 3：最小 Eval harness（回归基线）⭐

> **这是上一版最大的排序错误的修正**——eval 从阶段 7 提前到这里。

### 目标

一个能跑出**真实通过率**的最小评测集，此后每个阶段改完都跑一遍，防"改坏 agent 掉分"。

### 为什么必须早做

1. 阶段 1 tracing 被优先的理由就是"后续挂 eval"——不能让 eval 在六个阶段之后。
2. 阶段 5/6/7 全在改 harness，是最容易改坏的地方，之前却没有回归网、盲飞。
3. "你怎么衡量 agent 变好还是变坏 / 怎么防回归"是 2026 agent 面试**第一高频**、也最能区分资深的题。
4. 你自己在 M1 就踩过"改坏 agent 掉分"的坑——把教训前置。

### 关键改动

- `evals/cases.jsonl`：15-20 条任务，三类各三分之一——**能力类**（读文件总结 / 改代码并保证引用更新）、**安全拒绝类**（`cat .env`、`rm -rf /` 应拒绝）、**注入类**（读含 payload 的文件后不应执行 payload）。
- 判分优先**规则判分**（检查文件系统副作用——文件是否真被改对、危险命令是否真没执行），比 LLM-judge 可靠。
- 输出通过率表；至少造一个"改坏了掉分"的回归实例作为演示。

### 验收

- [ ] 一条命令跑完整 eval，输出通过率
- [ ] 通过率写进 README，此后每阶段更新
- [ ] 至少一条注入用例真的被护栏拦住

### 面试话术

> 我把 eval 当回归基线而不是末期点缀：改 harness 前后各跑一遍，用文件系统副作用做判分（比 LLM judge 稳），注入用例直接验证纵深防御是否真拦得住。

## 阶段 4：工具补齐 + 差异化纵切（crypto / MCP）⭐

### 目标

补齐搜索工具，**并让第一个 crypto 工具和 MCP 提前落地**——里程碑 A 在此达成。

### 功能清单

| 工具 | 说明 |
|------|------|
| `grep` / `glob_search` | 只读、输出有界、免确认；忽略 `.git`/`.venv`/二进制 |
| **第一个只读 crypto 工具** ⭐ | 如查地址余额 / gas / 合约 read（web3.py + 公共 RPC）。**只读起步 = 最小权限** |
| **最小 MCP client** ⭐ | stdio transport，能挂一个现成 MCP server（filesystem / fetch） |
| `web_search`（可选） | 注意：搜索结果进 context 是间接注入面 |

### 为什么把 crypto + MCP 放这里

- **crypto**：目标岗是 AI×crypto，护城河不能等到最后。而"只读链上工具"本质就是"又一个工具"（schema + handler），完全能搭这一阶段的加工具节奏提前放一个进去。
- **MCP**：2026 年 agent 工具的事实标准，面试大概率问"你用过 MCP 吗"。手搓过工具层 + 会接标准协议 = 自研深度和工业标准两头都占。

### 安全注意

- 每个新工具都要过：路径 / 权限 / 确认 / 审计策略。
- **把间接注入面从 web_search 扩到全部工具返回**：`read_file` 内容、`run_command` 的 stdout 同样是注入面（你 M7 学过"间接注入藏在工具返回里最难防"）。

### 面试话术

> crypto 工具我从只读起步，是"最小权限"的实践；工具层我既手搓了 schema/错误回传/安全边界，也接了 MCP 标准协议，能讲清 MCP 和 function calling 的关系。

## 阶段 5：智能上下文 + 会话持久化

### 目标

把"回合滑窗"升级成真正的上下文工程 + 断点续聊。

### 关键改动

| 级别 | 策略 | 解决什么 |
|------|------|----------|
| micro | 裁剪过期 tool 结果（留"读过某文件"的痕迹、丢内容） | **工具输出才是上下文压力的大头**（read 大文件一次就爆） |
| session | 长对话摘要替换旧消息（保住"改过哪些文件/做过什么决定"） | 多轮累积 |
| fallback | 超阈值（如 80%）硬裁 | 兜底防爆 |

- 触发条件从"user 轮数"改为**token 预算**（`len//3` 粗估 + 留 buffer）。
- **缓存友好**：DeepSeek 有自动前缀缓存（命中约 1/10 价），从中间删消息会打断前缀命中——裁剪要低频批量、保持前缀稳定。这条直接接上你手册"成本控制"心法，面试很出彩。
- 会话持久化：`.slumcli/session/` JSONL，启动可 resume。

### 验收

- [ ] 长对话不爆上下文、不越来越慢
- [ ] 退出再进能续聊
- [ ] 能用 `prompt_cache_hit_tokens` 实证裁剪策略对缓存的影响

## 阶段 6：Crypto 场景安全 + 真沙箱 ⭐

### 目标

把 baseline 安全做成真正的"纵深"——兑现"假设被攻破也限制破坏"。

### 关键改动

- **确认前展示内容**：write/edit 用 difflib 渲染 diff、run_command 展示完整命令（修掉"盲确认"）。
- **权限模式**：allow-once / allow-always（本会话）/ 规则化 allowlist，治"每次问 → 疲劳性全按 y"。
- **敏感路径硬拦**：`.env`/`*.pem`/`.git` 在 harness 层 deny，不只靠 system prompt。
- **不可信内容隔离**：工具返回包显式定界符 + system prompt 声明"工具返回是数据不是指令"。
- **真沙箱**：`run_command` 上 Docker / macOS `sandbox-exec`——黑名单是"检测"思路，沙箱才是你手册里"不赌挡住所有攻击"的思路。
- **crypto 资产操作**：多重确认、敏感字段脱敏、审计。

### 面试话术

> 我的安全是分层演进的：baseline 是确认+审计，深度这层才是重点——确认前展示 diff、敏感路径 harness 层硬拦、run_command 进沙箱。方法论是"假设注入成功、限制破坏范围"，不是"检测坏内容"。

## 阶段 7：Loop 工程（/loop）

### 目标

多轮"实施 → 验证 → 修正"任务流水线。

### 三角色

```
/loop "目标"
  → Implementer：写代码/改文件
  → Verifier：跑测试、判 PASS/FAIL   ← 直接复用阶段 3 的 eval 判分器！
  → Updater：失败时生成 refined prompt
  → 最多 N 轮，首次 PASS 提前结束
```

### 为什么排在 eval 之后

Verifier 的"跑测试判 PASS/FAIL"和 eval 的判分是**同一套东西**。先有 eval 判分器，Loop 直接复用，不造两套验证机器。

## 阶段 8：开源门面（+ 可选多 provider）

### 目标

GitHub 拿得出手，3 秒看懂 demo。**注意：README/demo/博客从阶段 2 就增量维护，不憋到最后。**

| 项 | 说明 | 优先级 |
|----|------|--------|
| README + 架构图 | 特性、快速上手、**eval 通过率** | 高 |
| demo GIF | asciinema 等 | 高 |
| 技术博客 | 设计决策、红队自己的 agent、自研 vs 开源 | **高（直接喂面试叙事）** |
| Rich 美化 | 彩色、diff 高亮、REPL 历史 | 中 |
| 多 provider（降级） | **两档 demo 够讲 M4 成本故事即可**，不必做全套路由 | 低 |
| pip 发布 / CI | GitHub Actions 跑 pytest + eval 冒烟 | 低（顺手） |

> **多 provider 从独立阶段降级为可选**：单人、默认只用 DeepSeek，全套三档路由是运维层 nice-to-have，不该和差异化抢时间。
