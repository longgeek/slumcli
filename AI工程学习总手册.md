## 2. 第一程:四大能力模块 ✓

### M1 · 旗舰项目(完整 agent)✓

手搓了一个完整 agent,覆盖:
- **工具调用 + agent 循环**:模型说要调工具 → 代码真去调 → 结果喂回 → 再想(这就是 harness 雏形)
- **RAG**(开卷考试):从 word-overlap 到 embedding 检索,踩过"检索决定上限/garbage-in-garbage-out"
- **embedding**(语义坐标):意思近 → 向量近
- **记忆**:跨会话存取(memory.json)
- **多 agent**:planner + worker,踩过"agent 间不共享上下文"
- **评估(EVAL)**:规则判分,踩过"改坏 agent 掉分"
- **可观测**:Tracer 记录每步
- **缓存**:TTL 权衡(实时不缓存、历史可缓存)
- **双护栏**(纵深防御):输入护栏 + 动作护栏(危险操作 human-in-loop)
- **FastAPI 服务化**

### M4 · 模型服务 ⭐(推理 + 部署)✓

**为什么用 GPU**:模型 = 海量简单乘加;CPU=少数全能博士(少而精),GPU=海量流水线小工(多而专)。海量简单计算要"多而专"。

**为什么贵**:参数要摊进显存(VRAM),大显存 GPU 又稀缺又贵。

**为什么慢**:memory-bound —— 瓶颈在"搬参数"不在"算"。每生成一个 token 都要重搬整套参数,所以逐字蹦。

**三招提速**(全冲着"搬运慢"这个根):
| 招 | 治什么 | 代价 |
|---|---|---|
| KV cache | 每字重算前文(平方级浪费)→ 攒着只算新字 | 占显存 |
| batching | 搬一趟只服务1人太亏 → 拼车多人共用 | 单延迟略增 |
| 量化 | 参数搬运量大 → 把参数压小(降精度) | 精度略降 |

**选型**:数据能出门+图省心 → 调 API;数据不能出门(金融/医疗)+ 要掌控 → 私有化部署。

**私有化部署六步**:租GPU → 装环境/驱动 → 下模型权重(HuggingFace/ModelScope) → ★推理引擎跑(vLLM/TGI/SGLang,自动做三招) → 暴露 OpenAI 兼容 API(agent 换 base_url 即可) → 生产化(Docker/K8s/Prometheus,你本行)。

### M7 · AI 安全 ⭐(差异化招牌)✓

**五大威胁**:输入(注入)、动作(越权)、输出(泄露)、数据隐私、红队。

**Prompt 注入**:
- 根源:LLM 分不清"指令"和"数据"(都是文字,靠理解区分,理解能被骗)
- 直接注入(用户打的,好防些)vs 间接注入(藏在文档/网页/工具返回里,用户无辜,最难防)
- 防御心法:**不靠检测坏内容,假设被攻破 → 限制破坏**
- 三招纵深防御:① 最小权限(被骗也干不了大事)② 不可信内容隔离(标记只读,但隔离指令本身也可能被绕过)③ 关键动作人工确认。每招都不完全可靠,所以叠加。

**数据泄露**:
- 注入是手段,泄露是目的;三种来源:被注入套出 / 越界检索 / 无意带出
- 防御:源头权限隔离 + 输出脱敏 + 最小数据暴露(治本=最小暴露,数据没进上下文就无所谓泄露)

**红队测试**:= 安全视角的评估。穷举攻击不可能 → 工具化大规模攻击(Promptfoo/Garak/PyRIT)→ 发现修复再测 → 上线。

**隐私合规**:数据全链路管控(从哪来/给谁看/存哪/留多久/何时删),金融刚需。

### M8 · 系统架构 ⭐(本行)✓

**核心**:做权衡决策,不是画分层图。

**方法论**:需求 → 抓约束 → 约束翻译成决策 → 拼架构 → 评估风险。

**关键认知**:"最重要的目标" ≠ "最该先定的约束"。按"锁死程度/推翻成本"排序,先定地基性硬约束(如"数据不外泄"一票锁死私有化),再处理打磨性目标(准、快)。

**实战**:完整设计了企业知识库问答系统,四约束全落地:
- 准 → RAG + 重排 + 生成防瞎编 + 评估
- 快 → 缓存 + 推理优化 + 模型分流
- 不外泄 → 私有化 + 整链路内网
- 权限 → 检索源头过滤 + 向量库权限元数据 + 鉴权拿身份

## 3. 第二程:原理冲刺 P1-P4

> 对着 slumcli(Claude Code 式 agent)项目,补做内核需要的原理。

### P1 · LLM 生成机制 ✓

- **逐 token 生成循环**:预测一个 → 加到末尾 → 重读全部 → 再预测。必须逐个(后字依赖前字),每轮重读全部(这是上下文/计费的根)。
- **token**:模型处理文字的基本单位(中文≈一字一token,英文一词可能拆几个)。为统一处理多语种而设计。
- **为什么对话越长越慢越贵**:输入token(历史+新问题)+ 输出token(逐个生成)。双重影响。
- ⚠️ **订正(面试高频追问点)**:有 KV cache 时**并非**"每 token 重算全部前文"。准确说——prefill 按输入长度算一次;decode 阶段每个 token 只需*读取*整个 KV cache(瓶颈是内存带宽,不是重算)。计费上历史每次请求都算 input tokens,DeepSeek 命中前缀缓存有折扣。按旧说法答"每字重读全部",面试官一句"不是有 KV cache 吗"就露怯。(与 M4 的 KV cache 段互相印证)

### P2 · 上下文工程 ★(JD 核心缺口,挖得最透)✓

- **上下文 = agent 唯一的工作记忆**:模型失忆,只能看眼前这张"纸条";纸条里有的能用,没有的等于不存在。
- **核心矛盾**:纸条有上限(按token算),但对话会一直变长 → 装不下。溢出会报错 / 越来越慢越贵。
- **四种管理策略**:截断(扔老的)、滑动窗口(只留最近N轮)、摘要(老对话压成总结)、压缩(去冗余)。
- **该留该丢(精髓)**:看两个维度 ——
  - **重不重要**(决定丢还是留)
  - **当前相不相关**(决定放纸条还是放向量库)
  - → 重要+要用=留纸条;重要+暂不用=存向量库(可召回);不重要=直接丢。
- **怎么知道分类**:大多不是"判断"是"记录"——
  - 大类:消息天生的 role 标签(system/user/assistant/tool),不用猜
  - 细类:agent 程序在信息产生时主动打标记
  - 难分的语义判断:才交给模型(最后手段,能不用就不用)
- **重要性判断用便宜模型**(模型分级思想):简单分类活用小模型,省钱。
- **核心智慧**:能"按需取回"就别"提前判断丢弃"(用 RAG 检索绕过"预判重要性"的难题)。
- **/compact**:主要是"摘要"——把长历史压成短摘要替换掉。

### P3 · Harness 工程 ★(JD 核心缺口,进行中)◐

- **harness 是什么**:包裹模型、让它能"做"的外壳。模型=大脑(只会想/说),harness=身体(眼/手/脚/嘴 + 循环节奏)。合起来才是能干活的 agent。
- **关键认知**:大脑(模型)大家都一样,**harness 做得好不好决定 agent 强不强**。Claude Code 厉害在 harness,不在模型特殊。
- **agent 循环(心脏)**:想→做→反馈→再想,一步一反馈。为什么不能一次想好全做完?因为大脑事先不知道会发生什么(支付会失败、店会关门),必须边做边看边调整。
- **循环控制两个坑**:
  - **停早了**:中间步骤完成误当任务完成(改完代码没测试就说搞定)→ 治:讲清"完成标准" + 加验证步骤
  - **停不下来**:死循环无限重试,烧钱 → 治:设最大循环次数 + 检测原地打转 + 失败到一定程度放弃求助
- **⬜ 还剩:工具体系设计**(给 agent 配的"手脚"怎么设计才好用)—— 待补
